# Cache Schema Bump 手順

このドキュメントは、ETag キャッシュスキーマバージョン (`SCHEMA_VERSION`) を上げる際の作業手順と注意事項を記載する。

関連: #230, #238, #239, #241, #254

---

## 1. いつバンプするか

`CacheEntry<T>` の構造（フィールド名・型・意味）が変わる場合にバンプする。  
**リクエスト挙動の変更だけ**（`conditionalRequests=false` 化など）では不要。

例:
- `expiresAt` フィールドの意味を変えた → バンプ必要
- `schemaVersion` チェックロジックを変更した → バンプ必要
- `fetchWithETag` のデフォルト引数を変えた → **不要**

---

## 2. バンプ手順

```typescript
// frontend/src/lib/cache/etagCache.ts
export const SCHEMA_VERSION = <旧バージョン + 1>
```

変更履歴コメントに以下を追記する:

```
 * - v<N>: PR #<PR番号>。<変更内容の一行説明>
```

---

## 3. Forward-only ルール（重要）

SCHEMA_VERSION は **必ず単調増加** させること。  
古いバージョンへ戻す（ロールバック）は **禁止**。

理由:
- 新しいクライアントが書いたエントリを古いクライアントが再利用してしまう恐れがある
- 「間違えて下げてしまった」場合は、さらに上のバージョンに上げて解消すること

---

## 4. デプロイ時のスパイク対策

バンプにより、デプロイ直後に全クライアントのキャッシュが無効化される（スキーマバージョン不一致による即時破棄）。

これは **想定内の挙動** であり、ほぼすべてのリクエストが 200 フルフェッチになる。

対策:
1. **バンプは通常デプロイと一緒に行う**（独立ロールアウト不要）
2. ETag キャッシュはフロントエンド専用のため、**backend 負荷は軽微**（DB クエリ結果は backend 側でキャッシュされる）
3. スパイクが懸念される場合は、バンプ前にバックエンドのインスタンス数を一時的に増やす
4. デプロイ後 5 分程度で全クライアントが新バージョンに移行し、キャッシュが再構築される

---

## 5. バンプ後の検証

```bash
# ユニットテスト: スキーマバージョン不一致で旧エントリが破棄されることを確認
cd frontend && npx vitest run src/lib/cache/etagCache.test.ts

# E2E: キャッシュ挙動の確認
cd frontend && npx playwright test e2e/cache-localstorage.spec.ts
```
